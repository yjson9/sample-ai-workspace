# OUTPUT_STANDARDS — 산출물 작성 표준

이 워크스페이스의 모든 산출물은 아래 표준을 따릅니다.

---

## 1. Markdown 문서 표준

- 모든 문서는 **UTF-8 Markdown**으로 작성한다.
- 문서 최상단에 제목(`# 제목`)을 둔다.
- 문단·목록·표를 활용하고, 한 문장이 지나치게 길어지지 않게 한다.
- 코드/명령/파일명은 인라인 코드(백틱)로 표기한다.
- 경로는 슬래시(`/`)로 표기하되, Windows에서는 역슬래시로 읽을 수 있음을 전제한다.

## 2. 표 작성 기준

- 표는 헤더 행을 반드시 둔다.
- 열 의미가 모호하지 않도록 헤더명을 명확히 한다.
- 빈 칸은 공백 대신 `-`로 표기한다.
- 신뢰도가 필요한 표에는 `구분`(확정/추정/확인필요/미확인) 열을 추가한다.

## 3. 파일명 규칙

- 기본: 영어 소문자 + 숫자 + 하이픈/언더스코어 사용, **공백 금지**.
- 권장 형태: `YYYYMMDD_주제_상태.md` 또는 `주제_vN.md`
  - 예: `20260714_api-spec_draft.md`, `db-schema_v2.md`
- 날짜가 의미 있는 산출물은 파일명 앞에 날짜를 둔다.
- 버전은 `v1`, `v2` … 로 표기한다.

**예외 1 — ID 접두사가 붙는 운영 문서** (요청/결정/질문/색인/검토/수정/승인 등):
- 대문자 ID 접두사 `REQ-`, `DEC-`, `Q-`, `DOC-`, `REV-`, `COR-`, `APP-` 를 **허용**한다.
  - `REQ-`=요청, `DEC-`=결정, `Q-`=질문, `DOC-`=문서, `REV-`=검토, `COR-`=수정, `APP-`=승인
- 주제 부분은 소문자 영문/숫자/하이픈을 **권장**하되, 팀 가독성을 위해 **한글 주제도 허용**한다.
  단, 공백은 금지하고 하이픈/언더스코어로 대체한다.
- 예(둘 다 허용): `REQ-001_api-doc.md`, `REV-001_api-문서화.md`, `APP-001_api-문서화.md`

**예외 2 — 템플릿 파일**:
- `_TEMPLATE_*.md` 형식(선행 언더스코어 + 대문자 `TEMPLATE` 표식)을 **허용**한다.
  복사해서 쓰는 양식임을 파일 목록에서 즉시 구분하기 위함이다. 예: `_TEMPLATE_request.md`.

> 참고: `01_inputs/`의 **원본 파일명은 제공된 그대로 유지**한다(추적성 우선).
> 공백·한글이 포함돼 있어도 임의로 바꾸지 않는다. 예: `단계별 수행업무 및 산출물.xlsx`.

## 4. 날짜 표기 규칙

- 날짜는 **`YYYY-MM-DD`** (ISO 8601) 형식을 사용한다. 예: `2026-07-14`
- 파일명에서는 구분자 없이 `YYYYMMDD`를 써도 된다.
- 상대 표현("어제", "다음 주")은 문서 본문에 쓰지 않고 절대 날짜로 적는다.

## 5. 상태값 규칙

상태값은 대상에 따라 **두 체계**를 사용한다. 대상 성격에 맞는 체계를 쓴다.

### 5-1. 작업·산출물 상태 (우리가 만드는 문서)
`03_working/`, `04_outputs/`의 작업물·산출물에 사용한다.

| 상태값 | 의미 |
|--------|------|
| `draft` | 작성 중 초안 |
| `review_requested` | 검토 요청됨 |
| `approved` | 사람이 검토·승인 완료 (승인 후에만 사용) |
| `rejected` | 반려됨(수정/폐기 필요) |
| `archived` | 보관됨 |

### 5-2. 입력자료 상태 (외부에서 받은 자료)
`01_inputs/`의 입력자료는 "작성 중/검토요청" 개념이 맞지 않으므로 별도 체계를 쓴다.

| 상태값 | 의미 |
|--------|------|
| `received` | 등록됨(수령). 아직 내용 검증 전 |
| `verified` | 내용을 확인·검증함(근거로 신뢰 가능) |
| `superseded` | 더 최신 버전으로 대체됨 |
| `archived` | 보관됨 |

> 입력자료를 `draft`로 표기하지 않는다("작성 중"인지 "미검증"인지 모호). `DOCUMENT_INDEX.md`도 이 구분을 따른다.

## 6. 리뷰 상태 규칙

- 산출물이 `04_outputs/`로 이동하려면 최소 1회 검토를 거쳐야 한다.
- 검토 요청 시 상태를 `review_requested`로 바꾸고 `05_feedback/review_requests/`에 요청서를 남긴다.
- 승인되면 `approved`, 반려되면 `rejected`로 갱신한다.

## 7. 분석 신뢰도 규칙

분석·판단에는 다음 신뢰도를 표기한다.

| 표기 | 의미 |
|------|------|
| `확정` | 근거로 확인된 사실 |
| `추정` | 근거로부터 유추, 미확인 |
| `확인필요` | 사람 판단/추가 정보 필요 |
| `미확인` | 근거 없음/정보 부족 |

## 8. 산출물에 반드시 포함할 섹션

모든 산출물 하단에 아래 메타 섹션을 포함한다.

```markdown
---
## 작업 메타
- 상태(status): draft | review_requested | approved | rejected | archived
- 작성일: YYYY-MM-DD
- 작업 목적:
- 사용한 입력자료:
- 판단 근거:
- 확정 사항:
- 추정 사항:
- 확인 필요 사항:
- 다음 작업:
---
```

## 9. 최종 산출물 vs 작업 초안

| 구분 | 작업 초안 (`03_working/`) | 최종 산출물 (`04_outputs/`) |
|------|---------------------------|------------------------------|
| 목적 | 분석·탐색·작성 과정 | 공유·전달 |
| 내용 | 추정·메모·미완성 허용 | 정제된 내용, 근거 유지 |
| 공유 | 내부용, 공유 대상 아님 | 검토 후 공유 가능 |
| 상태 | 주로 `draft` | `approved` |
| 추정 과정 | 그대로 남김 | 결론 위주로 정리, 확인 필요 사항은 명시 |
