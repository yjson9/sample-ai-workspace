# 재사용 프롬프트 — 산출물 검토 (review_output)

## 목적
AI(또는 사람)가 만든 산출물을 검토해 품질·근거·보안·형식을 점검할 때 사용합니다.

## 사용 방법
검토 대상 산출물(경로 또는 문서ID)을 지정해 아래 프롬프트를 AI에게 전달합니다.
검토 결과는 `05_feedback/review_requests/`에 저장합니다.

---

## 프롬프트

```
너는 지정된 산출물을 검토한다. 너는 승인자가 아니라 검토 보조자다.

[전제]
- 00_context/의 기준 문서와 관련 입력자료(01_inputs/), 요청서(02_requests/)를 참조한다.
- 검토는 사실 확인 위주로 하고, 불확실하면 확인필요로 표시한다.

[검토 대상]
- 산출물: {문서 경로 또는 문서ID}
- 관련 요청: {REQ-### 또는 없음}

[검토 기준] — 각 항목을 통과/미흡/확인필요로 판정하고 근거를 적는다.
1. 요구사항 반영 여부
   - 요청서/요구사항의 항목이 빠짐없이 반영되었는가
2. 근거 명확성
   - 주장·수치에 출처/근거가 있는가
3. 추정/확정 구분 여부
   - 확정/추정/확인필요/미확인이 올바르게 표기되었는가
4. 누락된 확인 필요 사항
   - 사람이 판단해야 하는데 표시되지 않은 부분이 있는가
5. 보안 위반 가능성
   - 개인정보/실데이터/접속정보/비밀값이 포함되지 않았는가
6. 산출물 형식 일관성
   - OUTPUT_STANDARDS(파일명/날짜/상태값/메타 섹션)를 따르는가
7. 다음 작업 가능성
   - 이 상태에서 다음 단계로 진행 가능한가, 막는 이슈는 무엇인가

[출력]
- 위 7개 기준별 판정과 근거
- 발견한 문제 목록(심각도 순)
- 수정 제안(구체적으로)
- 종합 판정(권고): approved 후보 / 수정 후 재검토 / rejected 후보

[하지 말 것] — 너는 승인자가 아니다.
- 산출물 상태를 approved로 바꾸지 않는다. 검토 단계에서는 review_requested를 유지한다.
- 05_feedback/approved/ 에 승인 기록(APP-###)을 만들지 않는다.
- 04_outputs/ 로 승격(이동)하지 않는다.
- 위 세 가지는 모두 '사람 승인' 이후에, 사람이 수행한다.
```

---

## 참고 (역할 경계)
- AI 검토의 산출물은 **판정 권고**까지다. 산출물 상태는 사람이 승인하기 전까지 `review_requested`로 둔다.
- **사람이 승인한 뒤에** 사람이:
  1. 승인 기록 `05_feedback/approved/APP-###_...`를 생성하고(템플릿 `_TEMPLATE_approval.md`),
  2. 산출물을 `04_outputs/{유형}/`로 승격하며 상태를 `approved`로 갱신한다.
- AI 권고가 "rejected 후보"면 사람 확인 후 상태를 `rejected`로, "수정 후 재검토"면 `03_working/`에서 수정한다.
