# 재사용 프롬프트 — 입력 삭제 후 정리 (clean_after_input_removal)

## 목적
입력자료(예: 엑셀)를 삭제한 뒤, 그로 인해 생긴 **파생물·참조**를 정리할 때 사용합니다.
기계적 삭제는 스크립트가, **판단이 필요한 문서 갱신은 이 프롬프트로 사람이 확인**하며 처리합니다.

## 사전 단계 (스크립트)
1. 사용자가 `01_inputs/`에서 입력 파일을 삭제한다.
2. 런처 `09_scripts/run.ps1` → **"입력 삭제 후 정리"** 를 실행한다.
   - `03_working/data/`의 고아 파생물을 미리보기 → 확인 후 삭제
   - `WORKSPACE_STRUCTURE.md` 재생성
   - 자료명을 '언급'하는 문서 위치 **리포트**(자동 수정 안 함)
   - **문서 참조가 있으면**, 아래 프롬프트의 `{경로/파일명}`·`{붙여넣기}`가
     **이미 채워진 완성본**을 `03_working/cleanup_request.md`로 생성한다.
3. `03_working/cleanup_request.md`를 그대로 AI에게 전달한다(빈칸 채울 필요 없음).
   AI에게는 "입력 삭제 정리해줘"라고만 해도 된다.

> 아래는 그 완성본의 **원본 템플릿**입니다(스크립트가 이 블록의 `{...}`를 채웁니다).
> 참조가 없으면 요청서를 만들지 않으며, 이 프롬프트도 실행할 필요가 없습니다.

## 프롬프트

```
너는 입력자료 삭제로 인한 참조를 정리하는 것을 돕는다. 너는 삭제 승인자가 아니며,
되돌리기 어려운 삭제·수정은 사람 확인을 거친다.

[입력]
- 삭제된 입력자료: {경로/파일명}
- 정리 스크립트 리포트(언급 위치 목록): {붙여넣기}

[해야 할 일]
1. DOCUMENT_INDEX.md
   - 삭제된 입력자료를 가리키는 행(위치가 그 파일)을 찾는다.
   - 원칙: 행을 지우기보다 상태를 `superseded`(대체됨) 또는 `archived`로 바꾸고
     비고에 "원본 삭제(YYYY-MM-DD)"를 남긴다. 완전 제거가 맞다면 사람에게 확인.
2. 그 자료에 '의존'한 작업물 점검 (03_working/analysis, 03_working/drafts, 06_decisions)
   - 근거가 사라져 재검토가 필요한 내용을 표시한다(자동 삭제하지 않는다).
   - 결정(DECISION_LOG)이 그 자료에 근거했다면, 결정을 지우지 말고 영향은 기록/질문으로 남긴다.
3. CHANGE_LOG.md
   - "입력자료 삭제 및 파생물 정리"를 한 줄 기록한다(대상/사유/영향/날짜).
4. 애매한 것은 OPEN_QUESTIONS.md 에 질문으로 남긴다.

[하지 말 것]
- '단순 예시 언급'은 수정하지 않는다.
  (예: OUTPUT_STANDARDS.md 의 파일명 규칙 예시처럼, 실제 파생물이 아니라
   규칙을 설명하려고 파일명을 인용한 경우 — 그대로 둔다.)
  각 언급이 (a) 실제 파생/참조인지 (b) 단순 예시인지 먼저 구분하고, (a)만 갱신한다.
- 04_outputs/(승인 산출물)는 사람 승인 없이 삭제하지 않는다.
- 원본이 아닌 파일을 근거 없이 지우지 않는다.

[마무리 보고]
- 갱신한 문서와 변경 요약
- 손대지 않은 '단순 예시' 목록(왜 그대로 뒀는지)
- 재검토가 필요하다고 표시한 작업물
- OPEN_QUESTIONS에 추가한 질문
```

## 참고
- 기계적 삭제(파생물)와 판단(문서 갱신)을 분리하는 이유: 파일명만으로 "파생물 vs 단순 예시 언급"을
  구분할 수 없기 때문입니다. 스크립트는 03_working/data의 고아 파일만 지우고, 나머지는 이 프롬프트로 처리합니다.
