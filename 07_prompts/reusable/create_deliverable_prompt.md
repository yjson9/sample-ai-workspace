# 재사용 프롬프트 — 최종 산출물 생성 (create_deliverable)

## 목적
`03_working/`의 내부 초안(또는 피드백을 반영한 초안)을 바탕으로 **공유 가능한 산출물 후보**를 만들 때 사용합니다.
(여기서 만든 후보는 아직 승인 전이므로, 이후 사람 검토·승인 단계를 거칩니다.)
후보는 `03_working/drafts/`에 `review_requested` 상태로 만들고, **사람이 승인한 뒤에**
`04_outputs/`의 유형별 폴더로 승격합니다(상태 `approved`).
(`04_outputs/`에는 승인된 산출물만 둡니다 — `WORKFLOW.md`, `DIRECTORY_GUIDE.md` 참조.)

## 사용 방법
원본 초안(경로)과 산출물 유형을 지정해 아래 프롬프트를 AI에게 전달합니다.

---

## 프롬프트

```
너는 지정된 초안을 바탕으로 공유 가능한 최종 산출물 후보를 만든다.
이 산출물도 사람이 최종 검토하기 전까지는 초안임을 잊지 않는다.

[입력]
- 원본 초안: {03_working/... 경로}
- 산출물 유형: {문서 / 스프레드시트 / 다이어그램 / 스크립트 / 리포트}
- 대상 독자: {예: 팀 내부 / 외부 이해관계자}

[출력 기준]
1. 공유 가능한 문체
   - 명확하고 간결한 서술. 내부 은어·임시 메모 제거.
2. 내부 추정 과정 제거
   - 탐색·시행착오 과정은 결론 위주로 정리한다.
   - 단, 결론의 근거(출처)는 유지한다.
3. 확인 필요 사항 명시
   - 아직 확정되지 않은 부분은 "확인 필요"로 명확히 표시한다(숨기지 않는다).
4. 출처와 근거 유지
   - 핵심 주장·수치에는 근거(입력자료/문서ID)를 남긴다.
5. 최종 검토자에게 필요한 정보 포함
   - 문서 하단에 OUTPUT_STANDARDS의 "작업 메타" 섹션을 붙인다.
   - 상태값은 review_requested로 설정한다. (승인 전이므로 approved가 아니다.)

[형식]
- OUTPUT_STANDARDS.md의 파일명/날짜/표/상태값 규칙을 따른다.
- 저장 위치: 03_working/drafts/ 에 만든다. (아직 승인 전이므로 04_outputs/에 두지 않는다.)

[하지 말 것]
- 승인 전 산출물을 04_outputs/에 만들지 않는다.
- 개인정보/실데이터/접속정보/비밀값 포함 금지. 예시는 가상 값만.
- 근거 없는 내용을 확정으로 서술하지 않는다.

[마무리 보고]
- 만든 산출물 경로(03_working/drafts/...), 상태값(review_requested),
  남은 확인 필요 사항, 검토 요청 대상, 승인 후 승격 위치(04_outputs/{유형}/)를 요약한다.
```

---

## 참고 (승인 흐름)
1. 이 프롬프트로 `03_working/drafts/`에 후보 생성 → 상태 `review_requested`.
2. `05_feedback/review_requests/`에 검토를 요청한다.
3. 사람이 승인하면 → `04_outputs/{유형}/`로 **이동(승격)**하고 상태를 `approved`로 갱신한다.
4. 반려되면 → 상태 `rejected` 또는 `draft`로 두고 `03_working/`에서 수정한다.
